# 에이전트 작업 흐름 (harness 엔지니어링)

> AI 에이전트의 표준 작업 절차. 사람이 다시 지시하지 않아도 같은 품질로 재현되는 것이 목표.
> 근거: Anthropic harness 엔지니어링(context engineering · gather→act→verify→iterate · generator≠evaluator) + Claude Code 베스트 프랙티스.

## 루프 — gather → plan → act → verify → iterate
모든 작업은 이 루프를 따른다. **매 행동 후 검증**하고, 막히면 무한히 돌지 말고 에스컬레이션한다.

## 컨텍스트 엔지니어링
1. **컨텍스트는 희소 예산** — 고신호 토큰만 올린다. 창이 찰수록 추론이 흐려진다(context rot). 큰 창보다 **가벼운 컨텍스트**가 신뢰도를 높인다.
2. **광범위 탐색·다중 파일 조사는 서브에이전트에 위임** — 서브에이전트가 격리 탐색 후 **결론만** 반환(원본 파일·로그 덤프 금지). 메인 스레드는 단일 파일·문서·대화에 보존.
3. **파일시스템을 기억으로** — 계획·진행상황·검증 결과를 파일로 남기고 필요할 때 당겨온다. 컴팩션에 기대 세션을 잇지 말고, 세션 경계는 **명시적 산출물**(진행 요약·검증된 테스트 결과·커밋된 git 상태)로 핸드오프.
4. 무관한 작업 사이에는 컨텍스트를 비운다(`/clear`), 긴 작업은 컴팩션한다.

## 검증 우선 (verification-first)
5. **읽지 말고 실행해 검증** — 에이전트가 읽을 수 있는 신호(typecheck/lint/test/build/스크립트)를 매 행동 후 돌린다. 정적 분석만으로 통과 선언 금지.
6. **객관 Hard Threshold로 완료 판정** — exit 0 · 테스트 100% · build SUCCESS 같은 pass/fail. "괜찮아 보임" 같은 주관 점수 금지.
7. **주장 말고 증거** — 실제 출력(`42/42 PASSED`·exit code·diff)을 제시.
8. **생성자와 평가자를 분리** — LLM은 자기 작업을 관대하게 평가한다. 검수는 새 컨텍스트/서브에이전트(예: qa-reviewer)로, 코드 정확성은 LLM 판단보다 코드 검증을 우선.

## 계획·반복·게이트
9. **계획과 실행을 분리** — explore → plan → implement → commit. 다중 파일 변경은 plan 모드/설계 문서 먼저. 3파일↑/다중 세션 작업은 `docs/plans/{날짜}-{slug}.md`에 저장.
10. **모든 루프에 유한 상한 + 에스컬레이션** — 예: "build 수정 최대 N회 후 중단/보고". 무한 루프·비용 폭주 방지.
11. **완료 기준을 먼저 선언** — "테스트 통과 AND 린트 0 AND 리뷰됨"처럼 "done"을 명시해 일찍 멈추거나 헛돌지 않게.
12. **가정을 먼저 명시** — 해석이 갈리면 구현 전 확인. **작업 단위 = 커밋 단위**("그 김에 고친" 코드는 별도 커밋).

## Claude Code 프리미티브 — 올바른 도구 선택
13. **skill / subagent / hook 구분** — *skill*(`.claude/skills/*/SKILL.md`)=재사용 절차(이름+설명로 점진 노출) · *subagent*(`.claude/agents/*.md`, frontmatter `description`/`tools`/`model`)=컨텍스트 격리 워커 · *hook*(PreToolUse/PostToolUse/Stop 등)=**매번 강제**돼야 하는 결정적 동작(자동 포맷·위험 명령 차단)은 CLAUDE.md 산문이 아니라 hook으로.
14. **always-on 컨텍스트를 최소화**(핵심 원칙) — "~200줄"은 숫자 목표가 아니라 *always-on을 작게 유지하라*는 원칙(context rot 방지). CLAUDE.md엔 **전 작업에 적용되는 헌법급 원칙만** 상시 두고(ccx: `ccx:always` 마커 룰만 @import), 프레임워크/툴 상세는 **on-demand**(색인만 두고 해당 영역 작업 시 읽기)로 분리. 권한은 deny-first + 도구/파라미터 글롭으로 좁힌다.

## DON'T
- 문서 갱신 없이 코드만 쌓지 않는다 · 로컬 검증 없이 push·PR 하지 않는다(`--no-verify` 금지) · 계획 없이 대규모 리팩터(5파일↑) 시작하지 않는다 · "작동한다"를 실행 없이 선언하지 않는다.

## 에스컬레이션
변경이 계획의 2배↑ / 기존 테스트 3개↑ 수정 / 외부 API 불확실 / 보안·권한·개인정보 / 동일 증상 2회↑ 실패 → 자동 진행 말고 보고.
