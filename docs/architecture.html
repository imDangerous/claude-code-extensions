ccx claude-code-extensions · AI 표준

Claude Code 프로젝트에 공통 AI 표준을 깔고, 특화 pack으로 보강하는 설치/관리 CLI.

🎯 목표 — “모든 프로젝트에 동일하게 적용되는 AI 표준 모델”을 만든다. core가 표준 베이스이고, web/app/spring 같은 특화 pack이 그 위에 보강된다. 값·도메인은 프로젝트가, 표준 구조는 ccx가 관리한다.

CLAUDE.md (그 외) · 도메인 문서project 직접 작성
설치 단위는 pack — ccx <pack> init이 그 팩 모듈을 일괄 설치. 모듈은 manifest targets로 “어디 깔리는지” 스스로 안다. init 질문은 팩 모듈의 합집합·중복 제거해 한 번씩만, 결과는 팩 1개 config.json에 저장.

3 · ccx apply — CLAUDE.md 2계층 주입 (always-on + 색인)

# CLAUDE.md (프로젝트 루트)
<!-- ccx:managed:start -->
## AI 표준 (ccx 관리 — 직접 수정 금지)
# always-on: ccx:always 마커 헌법급 원칙만
@.claude/rules/agent-workflow.md
@.claude/rules/validation.md
@.claude/rules/entropy.md
### 상세 규약 (작업 시 읽기 — always-on 아님)
- .claude/rules/nextjs.md — Next.js App Router
- .claude/rules/tailwind.md — Tailwind
<!-- ccx:managed:end -->

# ↓ 여기부터 프로젝트 소유 (자유 편집)
## 스택 / 도메인 / 용어

왜

  • 설치 = 자동 로드 아님. Claude Code는 CLAUDE.md와 그 @import를 읽는다.
  • 2계층 — <!-- ccx:always --> 마커(헌법급 원칙)만 @import(always-on), 나머지는 색인(경로+제목)으로 두고 해당 영역 작업 시 읽음 → always-on 컨텍스트 최소화(context rot 방지).
  • 관리 블록만 ccx 소유 → update가 블록만 갱신, 나머지 CLAUDE.md는 보존.
  • 서브에이전트·스킬은 @import 아님(자동 인식). doctor가 드리프트를 잡는다.

4 · 표준(ccx) vs 프로젝트 소유

🔵 ccx 관리 (표준, 직접 수정 ✗)

  • core/특화 pack의 룰·스킬·서브에이전트
  • commitlint·husky 훅·CI 워크플로
  • CLAUDE.md 관리 블록(@import)
  • 갱신: ccx … update · 점검: doctor

⚪ 프로젝트 소유 (직접 작성 ✓)

  • CLAUDE.md 진입점(스택 사실·설치 pack 요약)
  • config.json 값(티켓 prefix·브랜치·…) 예: { "ticketPrefix": "ABC" }
  • 도메인 문서 — 이 앱 고유(라우팅·플로우·용어집)
= “구조·표준은 ccx, 값·도메인은 프로젝트.” 이게 ccx가 **공용**일 수 있는 이유.

5 · 라이프사이클

# 설치(머신당 1회, 무인증)
curl -fsSL https://github.com/<owner>/claude-code-extensions/releases/latest/download/install.mjs | node

# 팩 단위 설치 — 한 번에 표준 전체
ccx core init              # core 팩 전체(모든 모듈)
ccx web  init              # web 보강 (core + js 자동 동반)
ccx apply                  # 설치 pack을 CLAUDE.md에 주입

ccx core doctor            # 팩 전체 점검
ccx core update            # 팩 전체 최신화(값·관리블록 보존)
ccx core update --only git  # (선택) 특정 모듈만 — 고급
